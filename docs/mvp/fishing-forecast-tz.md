# ТЗ: Приложение прогноза клева (MVP)

## 1. Цель
Разработать приложение прогноза клева для Красноярского водохранилища с поддержкой двух видов рыбы (`pike`, `perch`), 7-дневным прогнозом, журналом уловов и базовой эксплуатационной готовностью на сервере `fazendaserv`.

## 2. Границы MVP (In Scope)
- Backend API:
  - `GET /health`
  - `GET /ready`
  - `GET /forecast` (7 дней, виды `pike` и `perch`)
  - `POST /catch` (валидация и привязка к ближайшему погодному снимку)
- Data pipeline:
  - ежедневный сбор погодных данных;
  - получение температуры воды (fact + forecast);
  - локальный расчёт фазы луны.
- ML:
  - baseline-модель для `pike` и `perch`;
  - периодический retrain при достижении порогов качества/данных.
- Mobile:
  - экран прогноза;
  - экран погодных деталей;
  - журнал уловов с offline-очередью;
  - ограниченная карта заливов;
  - гостевой режим + авторизация для отправки улова.
- Ops:
  - dockerized деплой;
  - базовый мониторинг;
  - ежедневный backup БД.

## 3. Вне MVP (Out of Scope)
- Социальная лента и публичная карта уловов.
- Продвинутые модели (LSTM/TFT).
- Аппаратные датчики воды (ESP32/DS18B20 + MQTT) как обязательная часть релиза.
- Полноценный веб-клиент.
- Глубокая персонализация модели на пользователя.

## 4. Функциональные требования API

### 4.1 `GET /forecast`
- Назначение: получить прогноз на 7 дней.
- Query-параметр:
  - `species` (опционально): `pike` или `perch`.
- Поведение:
  - если `species` передан, вернуть 7 дней только для этого вида;
  - если `species` не передан, вернуть 14 записей (по 7 дней на каждый вид).
- Поля ответа (MVP):
  - `generated_at`
  - `days[].date`
  - `days[].species`
  - `days[].score` (0..5)
  - `days[].confidence` (0..1)
  - `days[].air_temp_c`
  - `days[].pressure_hpa`
  - `days[].water_temp_c`
  - `days[].moon_phase` (0..1)
  - `days[].stale`

### 4.2 `POST /catch`
- Назначение: записать улов.
- Требования к авторизации MVP:
  - обязательный header `x-user-id`;
  - без `x-user-id` возвращать `401`.
- Тело запроса:
  - `species`: `pike | perch`
  - `score`: 0..5
  - `latitude`: -90..90
  - `longitude`: -180..180
  - `note`: опционально, до 500 символов
  - `caught_at`: опционально, только timezone-aware datetime
- Поведение:
  - привязать запись к ближайшему погодному снимку по дате;
  - вернуть созданную запись с `id`, `created_at` и linked weather-полями.

### 4.3 Служебные endpoints
- `GET /health`: liveness.
- `GET /ready`: readiness API-процесса.

## 5. Нефункциональные требования (MVP)
- `GET /forecast` p95 <= 250ms (из кэша <= 30ms).
- API error rate < 1% в сутки перед релизом.
- После перезапуска сервера сервис поднимается автоматически.
- Бэкапы БД выполняются ежедневно, хранение не менее 30 дней.

## 6. Критерии качества и релиза
- ML quality gates:
  - MAE <= 0.80
  - RMSE <= 1.10
  - Spearman >= 0.45
  - Top-day hit-rate >= 0.60
- Release GO:
  - нет открытых P0/P1;
  - пройдены API/ML/mobile quality gates;
  - пройдены backup/restore и rollback rehearsal.

## 7. Юридический минимум
- Обязательные документы:
  - Privacy Policy
  - Terms of Use
  - Data Deletion Policy
- В приложении:
  - согласие на геоданные до отправки улова;
  - отдельные toggles для push и аналитики;
  - ссылки на policy/terms на экране авторизации и в настройках.

## 8. Текущее состояние реализации (на момент фиксации ТЗ)
- Базовый backend на FastAPI создан.
- Реализованы endpoints:
  - `GET /health`
  - `GET /ready`
  - `GET /forecast`
  - `POST /catch`
- В `POST /catch` добавлена валидация и привязка к ближайшему погодному снимку.
- Хранение уловов пока in-memory (этап MVP foundation, до интеграции PostgreSQL).

## 9. Этапы дальнейшей разработки
1. Интеграция PostgreSQL и Redis в compose.
2. Перенос `catch_records` из in-memory в БД.
3. Подключение ingestion погоды/воды и планировщика обновлений.
4. Добавление baseline ML-пайплайна и retrain-процедуры.
5. Реализация mobile-клиента с offline-очередью.
6. Подготовка legal-пакета и релизных артефактов.
